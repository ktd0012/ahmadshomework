Project 3 is to write a lexer and parser for a small, eccentric programming language called WhileProc—so called because it supports only simple procedures, while loops, conditionals, some operations on floating points numbers, and printing such numbers. Your job is to write a tokenizer and a top-down, recursive-descent parser for the grammar below. This involves writing just two methods: lex and parse.

lex(s) should take a string and return a list of strings representing the sequence of logical tokens in the input. There are four kinds of tokens in WhileProc: identifiers that start with a letter (any case) or underscore and then may have any alphanumeric or underscore characters following; numbers, which are either one or more digits (0-9) or zero or more digits followed by a period followed by one or more digits; keywords proc, if, else, while, print; and symbols, one of +-/*^,:=<{}();. The grammar below uses x and f to indicate some identifier terminal and n to indicate a numeric terminal. Whitespace includes line feeds, carriage returns, spaces, and tabs—none of these characters may appear within a token.

parse(toks) should take a list of tokens (list of strings) and return an AST node from syntax.py, properly encoding the input WhileProc program. You should use the following grammar, but may want to implicitly left-factor some productions (e.g., A), inline some productions (e.g., L, X), and turn some uses of right-recursion into a flat while loop (e.g., E,M). The AST you produce should have the usual precedence and associativity for operators as discussed in lecture (e.g., left associativity for subtraction, right associativity for exponentiation). In the grammar below, nonterminals are uppercase letters and terminals are either lowercase letters (for numbers, n, and IDs, x or f) or literal strings.

You can run public tests that are included in the starter, and if you want to write your own (it is highly recommended you do), just copy an existing test folder and modify it. The test script will then find your new test when you run local tests again. As on the previous project, your grade will be a reflection of how many tests you pass from a suite containing the public tests you’re provided, along with a number of secret tests. Passing all the public tests you’re given is no guarantee of receiving a high grade on the project—you should be writing your own tests and trying to break your own code.

Below are some general guidelines:

  • You do not need to understand exactly how the run and interp methods are working, and you will not need to modify them, but it would be a good idea to read syntax.py to understand what is expected as input to each constructor and to see how the WhileProc language works. A program is a sequence of 1 or more statements conjoined with semi-colons. A statement can be a while loop, a proc definition, a print statement, or a returned expression. Note that a semi-colon does not end a statement, but sequences them when there are more than one (think of it as an operator that says “run the first statement for its effects (variable assignment, print, etc), throw away its value, then evaluate and return the right-hand-side statement/expression”). An expression implicitly returns its value, and both procedures and numbers (all treated as floats) may be passed around as first-class values. Procedures are not closures that can encapsulate bindings as in Python and many other languages (don’t worry if you don’t know how closures work), but they may be passed to themselves to simulate recursion if desired, and cannot have variable arity.

  • The following example prints "3.0\n":

    proc three() { 1+2 };

    print three()

     

  • If the above example is passed to your lex function, it should return:

    ["proc", "three", "(", ")", "{", "1", "+", "2", "}", ";",

     "print", "three", "(", ")"]

     

  • If the above list of tokens is passed to your parse function, it should return:

    SeqStmt(

      ProcStmt(Var("three"), [], Plus(Lit("1"),Lit("2"))),

      PrintStmt(Call(Var('three'), [])))

    Try writing an AST manually at the REPL and then calling its .run() method (run returns a string of the AST’s output when interpreted—in this case, "3.0\n").

  • The following example sums a range of numbers and prints the result (n.b., handling comments is extra credit, see below):

    proc sum(n,s)

    {

        // Assignment can be chained (both i and sum are set to 0)

        // This statement is the same as sum := (i := 0)

        sum := i := 0;

        while i < n

        {

            i := i+s;

            sum := sum + i

            /* A trailing semi-colon here would be an error */

        };

        sum

    };

    n := 100;

    step := 0.5;

    print sum(n,step)

     

  • If the above example is passed to your lex function (sans comments if you are not doing the extra credit), it should return:

    ['proc', 'sum', '(', 'n', ',', 's', ')', '{', 'sum', ':', '=', 'i', ':', '=', '0', ';', 'while', 'i', '<', 'n', '{', 'i', ':', '=', 'i', '+', 's', ';', 'sum', ':', '=', 'sum', '+', 'i', '}', ';', 'sum', '}', ';', 'n', ':', '=', '100', ';', 'step', ':', '=', '0.5', ';', 'print', 'sum', '(', 'n', ',', 'step', ')']

     

  • If the above list of tokens is passed to your parse function, it should return:

    SeqStmt(

      SeqStmt(

        SeqStmt(

          ProcStmt(

            Var('sum'),

            [Var('n'), Var('s')],

            SeqStmt(

              SeqStmt(

                Assign(Var('sum'), Assign(Var('i'), Lit('0'))),

                WhileStmt(

                  LessThan(Var('i'), Var('n')),

                  SeqStmt(

                    Assign(Var('i'), Plus(Var('i'), Var('s'))),

                    Assign(Var('sum'), Plus(Var('sum'), Var('i')))))),

              Var('sum'))),

          Assign(Var('n'), Lit('100'))),

        Assign(Var('step'), Lit('0.5'))),

      PrintStmt(

        Call(

          Var('sum'),

          [Var('n'), Var('step')])))

    Try writing an AST manually at the REPL and then calling its .run() method (run returns a string of the AST’s output when interpreted).

  • What you’re being asked to write for this project is very similar to the coding we did in lecture to parse regular expressions. You will find it helpful to review this code before getting started on your WhileProc parser.

  • Each AST node in syntax.py supports a __str__ method so it can be converted back into a string. For example, str(Plus(Var("xyz"),Lit("1"))) yields "(xyz+1)". You may find this useful for testing/debugging. The string representation of an AST can even be passed to lex and parse to obtain the same AST again.

  • Write your own tests! Write lots of your own tests. Start early. Just ask me if you’re still unsure how a method should behave.

  • Three opportunities exist for passing some extra credit tests: you can support a more complete WhileProc language (shown below), along with C and C++ style comments (i.e., single-line comments starting with // and ending with a line feed, or traditional comments /* ... */), and precise error handling (properly return an ErrorMsg object defined in syntax.py whenever the input is a syntactically malformed program)—full credit only requires handling correctly formatted input.

    P -> S ";" P | S

    S -> "proc" f "(" L ")" "{" P "}"

       | "if" C "{" P "}" "else" "{" P "}"

       | "while" C "{" P "}"

       | "print" C

       | C

    L -> x "," X | x | Ɛ

    X -> x "," X | x

    C -> E | E "<" E | E "=" E

    E -> T M

    M -> "+" T M | "-" T M | Ɛ

    T -> F N

    N -> "*" F N | "/" F N | Ɛ

    F -> A | A "^" F

    A -> "(" C ")"

       | x ":" "=" C

       | f "(" R ")"

       | x

       | n

    R -> C "," Q | C | Ɛ

    Q -> C "," Q | C
